Lektion 1 - Einführung

Was ist Apache JMeter?

  • open source, 100% Java Lasttest-Werkzeug

  • Ursprünglich zum Testen von Web Applikationen

  • Testpläne zur genaueren Spezifikation

  • HTTP, HTTPS, FTP, SOAP, JDBC Requests möglich

  • Entwickelt von Apache Software Foundation

  • Alternativen:

    • StresStimulus

    • Gatling

    • LoadNinja (Kostenpflichtig)

Introducing Apache JMeter: an open source, 100% pure Java load test application

  • Performancetests auf Web Applikationen

  • statische und dynamische Ressourcen ansprechbar

  • Simulation starker Auslastung gegen

    • Server / Servergruppen

    • Netzwerk

    • einzelne Objekte

Protokolle

Performancetests mit verschiedenen Protokolltypen möglich

  • Web - HTTP, HTTPS (Java, NodeJS, PHP, ASP.NET, …​)

  • SOAP / REST Webservices

  • FTP

  • Database via JDBC

  • LDAP

  • Message-oriented middleware (MOM) via JMS

  • Mail - SMTP(S), POP3(S) and IMAP(S)

  • Native commands und shell scripts

  • TCP

  • Java Objects

GUI mode

  • IDE mit allen Features

  • einfache und schnelle Erstellung von Testplänen

  • recording, building, debugging

CLI mode

  • Ohne grafische GUI

  • Effizientes Ausführen der Lasttests

jmeter -n -t test.jmx -e -o result.jtl

HTML Reports

 

vollständige HTML Darstellung aller Ergebnisse

  • Tabellen

  • Graphen

Weiterhin …​

  • Export als Text (JSON, HTML, XML, …​)

  • Portierbarkeit durch 100% reinen Java Code

  • Full Multi-Threading Framework

  • Caching, offline analysis, replay test results

  • einfach erweiterbar

Lektion 2 - Elemente eines Testplans

Welche Elemente gibt es alles?
Wie können diese konfiguriert werden?

Testplan

  • Wurzel des Testbaums

  • Spezifiziert grundsätzliche Einstellungen

    • Umgebungsvariablen

    • Threadgruppen simultan oder einzeln laufen lassen?

    • Funktionales Testen (Sammeln aller Antworten anstelle selbst definierter)

 

Thread Group

  • Einstiegspunkte

  • Controller für die Threads

    • Anzahl an Threads (pro Thread eine Verbindung)

    • Ramp-up

    • Anzahl der Wiederholungen

  • Thread Groups arbeiten unabhängig voneinander

Thread Group

→ Guide: JMeter - Thread Group

Ramp-Up

Beispiel: 10 Threads, 100 Sekunden ramp-up

  • Start aller 10 Threads nach 100 Sekunden

  • Jeder Thread startet 10 Sekunden (100/10) nach dem vorherigen

 

Beispiel: 30 Threads, 120 Sekunden ramp-up

  • Start aller 30 Threads nach 120 Sekunden

  • Jeder Thread startet 4 Sekunden (120/30) nach dem vorherigen

Starten

  • Start: Selektierte Thread Group starten

  • Start no pauses: Selektierte Thread Group ohne Timer starten

  • Validate: Minimales Starten der Thread Group (1 Thread, 1 Iteration, keine Timer)

Variationen

  • setUp: Ausführung vor den Tests

  • tearDown: Ausführung nach den Tests

  • open model: Variable Anzahl an Threads (noch experimentell, → Dokumentation)

Samplers

  • Müssen unterhalb einer Thread Group definiert werden

  • Senden Requests, empfangen Responses

  • Der Reihenfolge des Auftretens im Testbaum nach ausgeführt

  • Gruppiert und einzeln konfigurierbar

  • Sammeln verschiedenste Ergebnisse:

    • Success/Fail

    • Antwortzeit

    • Größe der empfangenen Daten

    • …​

Verfügbare Samplers

→ Guide: JMeter - Samplers

HTTP Request Sampler

  • Sendet HTTP/HTTPS Requests

  • Dateien anhängbar

  • Vom Webserver empfangen:

    • Bilder

    • Java-Applets

    • Stylesheets sowie referenzierte Resourcen

    • Externe Skripte

    • frames, iframes

    • Hintergrundbilder (body, table, TD, TR)

    • Hintergrundsounds

HTTP Request Sampler - Limitierungen

  • Verhalten zu Connection re-use ungewiss

  • Kerberos Authentifizierung nicht möglich

  • Virtuelle Hosts nicht unterstützt

  • Nur GET, POST, HEAD, OPTIONS, PUT, DELETE und TRACE als Methode erlaubt

Logic Controllers

  • Müssen unterhalb einer Thread Group definiert werden

  • Steuern die Verarbeitung eines Tests

  • Definieren von Logik

  • Können die Ausführreihenfolge von Sampler ändern

  • Auch Loops möglich

Verfügbare Logic Controller

→ Guide: JMeter - Logic Controllers

Beispiel

Reihenfolge: Login → Load Search Page → Search "A" → Load → Search "B" [→ Load → Search "A" → …​]

Aufgabe 2.1: Sampler und Logic Controllers

Erstellt einen Lasttest gegen den echo-Webserver. Dabei soll folgendes eingehalten werden:

  • Es sollen bis zu 50 Nutzer simuliert werden

  • 0.2 Sekunden nach dem vorherigen Nutzer startet ein neuer Nutzer seine Anfragen

  • Jeder Nutzer soll sich einmalig anmelden (Simuliert durch localhost:5050/login)

  • Anschließend tätigt der Nutzer insgesamt 50 zufällig verteilte GET-Requests gegen die Ressourcen /hello und /world

  • Untersuchen Sie weitere Sampler / Logic Controller auf ihre Funktionsweisen.

 

Hinweise:

  • Fügen Sie der Thread Group zur Betrachtung der Ergebnisse einen Summary Report hinzu (add → Listener → Summary Report)

  • Versuchen Sie fernen, den Test über das CLI zu starten

  • Bislang tätigt jeder Nutzer seine 51 Anfragen und hört dann auf. Wie kann dafür gesorgt werden, dann dieser Nutzer immer von vorne beginnt?

Listeners

  • Bereitstellung von Informationen

  • Auf beliebiger Ebene

    • Testplan-Ebene: Sammeln der Daten von allen Sampler

    • Thread-Group-Ebene: Alles, was der Thread Group zugehörig ist

    • …​

  • Speicherung als CSV oder XML möglich

Verfügbare Listeners

→ Guide: JMeter - Listeners

Ergebnispräsentation

Alle Listener sammeln alle Daten von den Sampler ein. Einziger Unterschied: die Repräsentation

  • Graph Results: Plotten der Ergebnisse als Graph

  • View Result Tree: Details zu Sampler Requests und Responses als HTML oder XML

  • Summary Report: Stochastische Kenngrößen zu Sampler Requests und Responses

Performance

Viele Listener sind aufwendig

→ Hoher Speicherverbrauch

→ Hohe CPU Auslastung

→ Verfälschte Ergebnisse

→ Besser: Abspeichern der Ergebnisse, Darstellung hinterher (Simple Data Writer)

Timers

  • Pause vor Samplers im Scope

  • Simulieren "Bedenkzeit" von Nutzer

  • Addition bei mehreren Timer

  • Beispiele:

    • Constant Timer: Festgelegte Zeit (default 300 ms)

    • Uniform Random Timer: Gleichverteilt zufällig

    • Gaussian Random Timer: Normalverteilt zufällig

    • Synchronizing Timer: Warten, bis x Threads warten

→ Guide: JMeter - Timers

Assertions

  • Asserts auf Responses nach Sampler im Scope

  • "Testen" auf Richtigkeit

  • Auch direkt einem Sampler unterordbar

  • Beispiele:

    • Response Assertion: Check auf Patterns

    • Size Assertion: Länge des Responses

    • HTML Assertion: HTML Syntax

  • Assertion Listener, einsehen der Ergebnisse

→ Guide: JMeter - Assertions

Configuration Elements

  • Modifikation von Request

  • Hierarchie: Spezifische Configs überschreiben generische

  • Typische Anwendung:

    • Generisches Setzen einer Ziel-URL

    • Setzen von HTTP Header

    • Setzen von Cookies

    • JDBC konfigurieren

→ Guide: JMeter - Assertions

Configuration Elements: HTTP Request Defaults

 

  • Request Defaults

  • Alle Sampler nutzen hier definierte Einstellungen (wenn nicht spezifischer definiert)

Configuration Elements: HTTP Cookie Manager

 

  • Setzen von Cookies

  • Alle Sampler nutzen hier definierte Cookies (wenn nicht spezifischer definiert)

Aufgabe 2.2: Listeners, Timers, Assertions und Configuration Elements

Erweitern Sie den vorherigen Lasttest um folgende Funktionalitäten:

  • Jede /hello und /world Request soll in 5er Gruppen gestellt werden (Timeout 100ms)

  • Fügen Sie passende Configurations hinzu, um im Falle einer Adressänderung (z.B. anderer Port) nicht alle Sampler anpassen zu müssen

Lektion 3 - Anwendungsbasierte Testpläne erstellen

Wie sehen komplexere Testpläne aus?
User-Login, Sessions, LDAP, …​?

Testplan: User-Login

  • Viele Ressourcen nur als angemeldeter Nutzer zugänglich

  • Weboberfläche: Formular

HTTP Authorization Manager

Testplan: Sessions

  • 2 Möglichkeiten basierend auf die Applikation:

Testplan: Sessions - Session-Cookies

HTTP Header Manager

  • HTTP URL Rewriting Modifier

  • Name der Session ID angeben

  • JMeter erkennt diese automatisch

Lektion 5 - Best Practices zu JMeter

Worauf sollte immer geachtet werden?

Best Practices (1/4)

Anzahl der Threads geschickt wählen

  • Anzahl der Threads stark an Hardware gekoppelt (Server, Testrechner)

  • Zu wenige Threads: Wenig aussagekräftige Lasttests

  • Zu viele Threads: Abarbeitug der Anfragen kommt nicht hinterher → Coordinated Omission

  • Bei large-scale Tests: Weniger Threads mit mehreren CLI Jmeter Instanzen auf mehreren Maschinen.

 

Geschickte Platzierung von Cookie Manager

  • Unterhalb der Thread Groups

    → Jeder Thread hat eigene Cookies

    → Unter den Sampler teilen sich die Cookies

 

Geschickte Platzierung von Authorization Manager

  • Wie der Cookie Manager

Best Practices (2/4)

HTTP(S) Test Script Recorder verwenden

  • JMeter erkennt browsing durchs Internet

    → JMeter erstellt automatisch den Testplan

 

Mehrere Benutzer als CSV abspeichern

  • Textfile mit user names und passwords, getrennt durch Komma

  • CSV DataSet Configuration, Werte auf USER und PASS setzen

  • In Sampler: ${USER} und ${PASS} nutzen

 

Tests parametrisieren

  • Beispiel: Anzahl der Threads durch einen Parameter festlegen anstatt hard codieren

Best Practices (3/4)

Verwendete Resourcen minimieren

  • CLI mode benuzten: jmeter -n -t test.jmx -l test.jtl

  • Weniger Listener benutzen

  • Aufwendige Listener nur zu Debug-Zwecken

  • Besser Loops und Variablen statt viele ähnliche Sampler

  • CSV Output statt XML Output

  • Wenige Assertions verwenden

 

Properties-File managen

  • keine Properties in jmeter.properties ändern

  • Besser: Kopie der Propertie in user.properties einfügen und dort anpassen

 

JSR223 Elemente verwenden

  • Performancereichste Skriptsprache zu JMeter

Best Practices (4/4)

Aktuellste JMeter Version verwenden

  • Performance von JMeter wird stetig verbessert

  • Patch Notes informieren über Verbesserungen und neue Elemente

 

Keine veralteten Elemente nutzen

  • Veraltete Elemente haben oftmals geringere Performance

  • Werden durch Neue ersetzt

 

Variablen zwischen Threads und Thread Groups teilen

  • Variablen sind normalerweise an Thread gebunden

  • Speichern als Propertie oder in externes File zum sharen
